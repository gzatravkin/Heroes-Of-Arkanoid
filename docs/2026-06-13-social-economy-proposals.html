1Research digest — what the references actually do

Three reference points, distilled to the mechanics that transfer to a brick-breaker RPG.

🗼 The Tower

Idle/active endless tower-defense; deep meta-progression.

Power core

  • Cards — a large pool; you equip a limited number (slots unlock over time). Each is a passive modifier (damage, health, cash bonus, free upgrades…). Leveled by playing / spending duplicates.
  • Modules — 4 slots (Cannon / Armor / Generator / Core). Each has a rarity (Common→Ancestral), a main effect + rerollable substats, leveled with module shards.
  • Lab — time-gated permanent research.

Social / competitive

  • Weekly tournaments in bracketed leagues (Wood→Champion) with promotion / demotion; placement pays Medals/Stones/Gems.
  • Daily missions, milestones, guilds.

Steal: Cards+Modules as the persistent power layer; weekly promo/demo leagues.

⚔️ We Are Warriors

Casual army-merge battler; aggressive live-ops.

Events

  • Rotating themed events (a new era/army), each with its own event currency and a reward track (battle-pass shape, with a free lane).
  • Time-limited → fresh goals every cycle; event leaderboards; cosmetic + power payouts.
  • Daily login, spin wheel, ad-boosts.

Steal: a rotating event with a reward track is the cheapest way to keep a small game feeling alive between content drops.

🎴 StS / Hades our DNA

The roguelite spine the game already has.

  • High-variance runs; “every choice could go a different way.”
  • Permanent meta-unlocks that persist past a failed run (we already do this: Shards drip on death, boss-clears unlock spells/characters).
  • Run-draft build (relics / ball-cores / paddle-mods / spell picks) — already shipped.

Keep: the run is the fun. Cards/Modules must feed runs, not replace the moment-to-moment.

The transfer thesis. Our game is a deterministic, server-replayable brick-breaker RPG. That makes it a near-perfect fit for The Tower’s persistent Cards+Modules grind and for a provably fair competitive ladder — two things idle games fake with trust and we can do for real (see §3). We Are Warriors events are the low-cost retention glue on top.

3The shared spine — leaderboard, anti-cheat & the season clock

This part is identical across all three proposals — it’s the technical contract, not a design fork. It’s where your “abstract the provider” and “shadow-ban impossible scores” requirements live.

3.1 · Our secret weapon: a deterministic sim = a provably fair ladder

Arkanoid.Core is a pure, deterministic simulation (same seed + same inputs ⇒ same result, by design). So a leaderboard score doesn’t have to be trusted — it can be recomputed. The client submits the seed + the input log, never just a number. The server re-runs the headless sim and derives the canonical score. A score that can’t be reproduced is, literally, impossible → it never enters the board and the account is flagged. Most mobile leaderboards can only guess at cheating; we can prove it.
SUBMIT-SCORE (server-authoritative, runs on the leaderboard backend) ┌──────────────────────────────────────────────────────────────────────┐ │ client ──▶ { playerId, seasonId, seed, inputLog[], clientScore, │ │ buildHash, clientVersion, nonce, signature } │ │ │ │ 1. AUTH verify signed session token + nonce (no replays) │ │ 2. REPLAY serverScore = Arkanoid.Core.simulate(seed, inputLog) │ │ 3. MATCH |serverScore − clientScore| ≤ ε ? else → FLAG │ │ 4. PLAUSIBLE apm, duration, score/tick, inputs-in-bounds → FLAG │ │ 5. INTEGRITY buildHash ∈ allowed, version ≥ min → REJECT │ │ 6. WRITE upsert max(serverScore) into league bucket │ │ 7. SHADOW if player.shadowed → write to private shadow board only │ └──────────────────────────────────────────────────────────────────────┘

3.2 · Provider abstraction (Firebase today, anything tomorrow)

The game and server depend only on an interface. Firebase/Firestore is one adapter; you can swap to Supabase, PlayFab, a self-hosted Postgres, or Nakama without touching game logic or UI.

// One port, many adapters. The game never imports "firebase".
interface LeaderboardProvider {
  submitScore(s: ScoreSubmission): Promise<SubmitResult>;   // returns {accepted, rank, shadowed?}
  getLeague(seasonId, leagueId, page): Promise<LeaderboardPage>;
  getMyStanding(playerId, seasonId): Promise<Standing>;
  getSeason(): Promise<SeasonInfo>;            // id, startsAt, endsAt, resetCadence
  subscribeLeague(seasonId, leagueId, cb): Unsub;   // optional live updates
}

// Adapters implement the port. Selected by config, never referenced directly.
class FirebaseLeaderboard  implements LeaderboardProvider { /* Firestore + Cloud Functions */ }
class InMemoryLeaderboard  implements LeaderboardProvider { /* tests / offline dev */ }
class SelfHostedLeaderboard implements LeaderboardProvider { /* our .NET server + Postgres */ }
Where verification runs matters. Score replay must run on trusted server code — a Firebase Cloud Function that hosts the same Arkanoid.Core rules (compiled to WASM, or a tiny .NET re-impl), or our existing Arkanoid.Server acting as the verifier with Firestore as pure storage. Firestore alone can’t replay; keep the trust boundary on code you control.

3.3 · Shadow-ban — quiet, not loud

LayerCatchesResponse
Replay mismatchfabricated / edited scores (the “impossible” case)hard reject + strike
Plausibility boundstampered inputs (impossible APM, sub-second clears, score/tick over physical max)strike
Statistical outlierscores N σ above the seed’s distributionreview queue
Rate / noncereplay & spam submissionsthrottle

Past a strike threshold the account is shadowed: it still sees a normal-looking board, but it is served only its own entries (or a fake cohort), and its scores never reach real players. No error, no message — cheaters who get told they’re caught just iterate. Shadow state is a server flag, reversible, appealable, and logged.

Rule: never surface anti-cheat feedback to the client. The honest player never sees any of this; the cheater never learns what tripped.

3.4 · The season clock (drives every reset)

One authority defines time, so dailies, the league ladder, and events never drift apart:

SEASON (the unit of competition) e.g. 4 weeks └─ WEEK (league promo/demo + daily-pool reset) └─ DAY (daily missions reset 00:00 UTC) EVENT (overlaps a season, own cadence) e.g. 7–10 days, themed

Server-owned timestamps only (never the device clock — that’s a free exploit). The client reads getSeason() for endsAt and renders countdowns.

4Cards & Modules — the Tower-style power core

The persistent power layer the economy is being remade around. Shown once here; each proposal dials how much of it ships.

🎴 Cards passive, equip-limited

A growing pool of passive modifiers. You equip a few (slots unlock with account level / Shards). Tension = which few.

Examples (reuse our content vocabulary)

  • Molten Core — +X% ball damage.
  • Tithe — +X% Gold from coins.
  • Overflow — start each level with +Y mana.
  • Second Wind — first lethal hit per level is survived.
  • Draftsman — +1 spell-pick option in dungeons.

Progression

  • Drop as duplicates → Card Dust levels them (bigger %).
  • Sources: dailies, league chests, event track, dungeon clears.

Optionally absorbs Skill Points: “spell level” becomes a per-spell card. Fewer systems, one upgrade verb.

🧩 Modules slotted, rarity, reroll

Four slots mapped onto our four build axes — so Modules deepen the identity we already have:

🔵 Coreball-core effects (heavy, split, ember…)
🟫 Paddlepaddle mods (wide, grip, cannons…)
🔴 Balldamage / speed / crit substats
🟣 Fieldeconomy / mana / drop-rate substats

Rarity & substats

  • Common → Rare → Epic → Legendary → Mythic. Epic+ carries a unique named effect.
  • Rerollable substats using Module Cores — the long-tail chase.

Run-draft relics/cores/mods stay the per-run layer; Modules are the permanent layer that makes runs start stronger.

Fidelity guardrail (per CLAUDE.md): Cards/Modules are content. Each must be judged by how it plays vs the design docs, tested for its real trigger/identity, and not collapsed into a generic “+stat” bag. The slots map onto axes that already exist so nothing bespoke gets flattened.

5Proposal A — “Weekly Trial Ladder”

LeanNo new game modeCards only (Modules later) Build: ~2–3 wks

Smallest cohesive system that delivers all three pillars. Best first step; everything else can grow out of it.

Leaderboard

Weekly Trial: one shared seed per week — everyone plays the same hand-tuned gauntlet level. Score = blocks + depth + time bonus. Same seed ⇒ fair, and replay-verifiable (§3.1).

Leagues: 30-player cohorts, Wood→Champion. Top 7 promote, bottom 7 demote each week.

Rewards

Dailies: 3/day from a rotating pool that reshuffles weekly (“destroy 250 blocks”, “clear a dungeon floor”, “land 15 perfect deflects”). Pay Gems + Card Dust; a 7-day streak chest.

League chests by placement (Gems, Card Dust, rare Card).

Event: one monthly biome-modifier event w/ a short token track.

Economy

Merge Crystals + Campaign Gold → Gems. Keep Gold, Shards. Add Cards + Card Dust as the meta-power layer. Modules deferred.

Skill Points: keep as-is (don’t fold yet).

Why it’s “integral”: dailies mint Card Dust → Cards make Trial runs stronger → Trial placement mints more Dust + Gems → Gems buy the run-shop edge. One loop, three pillars.

6Proposal B — “The Spire”

Full Tower coreNew endless modeCards + Modules Build: ~6–9 wks

The most faithful to your The Tower reference, and the highest retention ceiling. Biggest economy remake.

New mode: Endless Gauntlet

Infinite escalating waves of blocks + periodic bosses; push for a high score. This becomes the main grind loop and the primary leaderboard — exactly The Tower’s endless run.

Per-run Gold + Module Cores scale with depth.

Leaderboard & leagues

Weekly tournament on Endless score, full bracket ladder, promo/demo. Placement → Medals → a league shop (exclusive Modules + cosmetics).

Replay-verified (§3); Endless is deterministic per weekly seed.

Economy (full)

Gold (run) · Gems (meta) · Shards (prestige) · Card Dust · Module Cores · Medals · Event Tokens. Skill Points fold into Cards.

Cards (equip 5, slots grow) + Modules (4 slots, rarity, reroll) fully in.

Why it’s “integral”: Endless feeds Module Cores → Modules + Cards raise your Endless ceiling → weekly tournament converts skill into Medals → league shop sells the next Module. Dailies + events are the off-ramps that keep non-tournament days rewarding.

Cost flag: Endless needs new pacing/balance + boss cadence, and the economy remake touches every meta screen. Highest payoff, highest risk — do it after A proves the loop.

7Proposal C — “Season Festival”

Live-ops firstWe-Are-Warriors-leaningCards + Modules-as-chase Build: ~4–6 wks

Wraps the existing campaign + dungeon in a recurring Season: a themed event, a reward track, and a ladder — content-ops as the engine.

The Season (4–6 wks)

Each season = a theme (e.g., “Inferno Rising”) with a reward track (free lane + optional premium lane), seasonal modifiers across all modes, and a season-end payout.

Leaderboard

Season Score = cumulative points earned across campaign + dungeon + event play, snapshotted into weekly league standings (promo/demo). Rewards both grinders and skilled players.

Rewards & economy

Daily board feeds Season Tokens → the track. Cards (moderate) as meta; Modules are the season-track chase items. Currencies: Gold / Gems / Shards + Season Tokens.

Why it’s “integral”: everything you already do earns Season Tokens → the track dispenses Cards/Modules/Gems → those power the next season’s ladder. The leaderboard isn’t a side mode; it’s a readout of the season you’re already playing.

Cost flag: low code risk, high content-ops commitment — a festival that stops rotating dies loudly. Best if you can commit to a content cadence.

9Recommendation — a phased path, not a pick

Ship A first; let it grow into B. Treat C as the content-ops wrapper you switch on once the systems exist.

  1. Phase 1 — Proposal A. Build the shared spine (§3): provider-abstracted leaderboard + replay-verification + shadow-ban + season clock. Add the Weekly Trial, leagues, dailies, and Cards. Merge Crystals+Campaign Gold → Gems. This is the smallest thing that exercises all three pillars and de-risks the hard part (anti-cheat) early.
  2. Phase 2 — grow into B. Add Modules (the four axes already exist), then the Endless Gauntlet as a second leaderboard. Fold Skill Points into Cards. The ladder you already built just points at a new score source.
  3. Phase 3 — switch on C. Once Cards/Modules/events exist, wrap them in recurring Seasons with a reward track. Now your live-ops calendar drives everything with no new core code.

Each phase is independently shippable and player-visible. The economy is remade incrementally (Gems merge first, chase currencies as their systems land) rather than in one risky cutover.

Worth confirming before we commit to a pure-Firebase backend.
