1. God classes / god methods
GameInstance is the central problem. It's not a class — it's a mutable state bag with ~70 fields that twenty static "systems" reach into directly. It accumulates per-feature state with no ownership boundaries: boss state (_bossFistCol, _goblinAnchorIdx, _seraphSummonVase), per-spell state (_phoenixRemaining, _lastDayCooldown, _magnetRemaining, _penetrationArmed), relic counters (_killsSinceSplit, _secondWindUsed, _hellwalkerUsedThisServe), power-up flags, combo counters, six separate ID counters. Every new mechanic adds 2–4 fields here. The underscore prefix is a lie — g._log, g._nextHazardId++, g._comboCount = 0, g._widePaddleActive are all mutated externally (BallSystem, CheatHandler, Snapshot). The "systems" pattern is ECS-shaped but without the E or the C: there are no components, just one god-entity whose internals everyone pokes.
BlockDamage.DamageBlock is the god method. One ~90-line function owns: HP subtraction, death, mana grant, combo advance, crystal payout, ball speed escalation, a hardcoded TypeId→power-up drop table ("hell_tough", "village_cauldron"...), fire spread, decay spread, bomb chaining, vase level-up, cauldron refund, lava retraction, two relic counters, and necromant revive queuing. Every feature that reacts to a block dying must edit this function. This is the strongest argument in the codebase for an OnBlockDestroyed event/hook pipeline — NecromantSystem.OnBlockDestroyed already shows the shape; everything else should follow it.
SimConfig — ~200 flat knobs in one class. No grouping per feature, so unrelated systems share a namespace (ShieldCost is the Paladin spell, ShieldDuration is the statue buff, ShieldStatueInterval is the enemy cadence). It works at this size; at 2× the content it becomes unsearchable. Nested config records per feature (config.Spells.Fireball.Cost) would also let Modifiers stop being a second balance-logic dumping ground.
Block is a union type pretending to be a class. 18 behaviors share one class carrying the superset of all parametric fields (TeleportColor, EmitInterval, StoredMana, FuseTimer, OwnerId, StatueLevel...). The comment says "no stored-bool soup," but it then defines 17 derived bool properties, and IsStatue => Emitter && EmitAim == "paddle" || ShieldStatue (which, note, parses as (Emitter && EmitAim=="paddle") || ShieldStatue — correct only by luck of intent). Behavior-specific data wants to live in behavior-specific components or a BlockBehaviorState object, not in every block.
2. Duplicate logic

Seven copy-pasted catalogs. BlockCatalog, BonusCatalog, CampaignCatalog, CharacterCatalog, DungeonCatalog, ItemCatalog, RelicCatalog all repeat Dto / FromJson / FromFile with inconsistent lookup semantics: BlockCatalog.Get throws a raw dictionary KeyNotFoundException, DungeonCatalog.Get throws a custom message, CharacterCatalog.Get does FirstOrDefault + throw (O(n)), some have TryGet, some don't. One generic Catalog<TDef> kills ~150 lines and unifies error behavior.
The spell-cast gate is copied ~20 times. if (Phase != Playing) return; if (Mana < cost) { log denied; return; } Mana -= cost; appears in every cast. The fix already exists — Spend() in the KitSpells partial — but the older FireMage/ClassSpells casts never adopted it. Same for the RaiseEvent("spellCast", ...) tail.
SpreadFire and SpreadDecay re-implement block death (nb.Hp -= chip; if (Hp<=0) Dead = true; RaiseEvent; mana) instead of calling DamageBlock. This isn't just duplication — it's a behavioral fork: kills via burn/decay skip combo, crystals, drops, relic counters, cauldron refunds, necromant marks, and (for fire) even the kill-mana grant. The two code paths will silently diverge with every new death effect.
wide_paddle vs powerup_wide in BonusSystem.ApplyEffect are the same ~12 lines differing only in which duration constant they read. Same family duplication between spawnBonus/spawnPowerUp cheats and SpawnWithType.
Bat-carrier and witch-grab "pop" logic in CombatSystem duplicate the rescue check, ball release with random lean, and GrabberId = 0 reset.
The drain line Grid.Height + CellSize * 2 is computed independently in UpdateBonuses, UpdateHazards, and WinLoseSystem. That's an invariant, not a local expression.
ProfileStore and DungeonStore are the same class twice (path-for-pid, lock, tmp-file atomic write).
ItemEffects.ApplyOne special-cases item IDs inside a switch on effect type (def.Id == "torch", "staff", "clover", "orb", "hourglass"). The whole point of a data-driven ItemDef is defeated; magnitude belongs in the JSON ("magnitudePerTier"), not in code that must know every item's name.

3. Leaky abstractions

Snapshot.From reads sim internals (g._widePaddleActive, g._comboMultiplier, g._bricksDestroyedThisLevel). The network DTO layer is welded to private state; you can't refactor GameInstance without breaking serialization. GameInstance should expose a read-only view, or systems should publish into the snapshot.
GameInstance mutates LevelData. Blocks => Level.Blocks means the "level definition" is also the live mutable game state — DungeonService.ApplyTier hardens blocks in the loaded level object, descend kills blocks in it, lava adds blocks to it. The instance must deep-copy blocks at construction; LevelData should be an immutable template.
DungeonCatalog.Register is mutated at request time. RiftService.GenerateRift registers rift-{biome} into the startup singleton catalog shared by all profiles, with no lock. Two players clearing levels concurrently race, and player A's tier-3 rift definition silently replaces player B's tier-0 one before B calls /dungeon/start. Generated rifts belong in the per-profile DungeonRun, not the global catalog.
Projectile is four things. Player projectiles, enemy hazards, bat carriers, and carts all share one class discriminated by Kind strings ("bat", "witchgrab", "stalactite", "cart", "allybolt") plus union fields (CarriedBallId, StateTimer, HitBlockIds). String-keyed dispatch is scattered across CombatSystem, BallSystem, and the renderer contract. Worse, EmitterSystem.FireAllyBolt adds to g.Projectiles using g._nextHazardId++ — the two ID sequences are already cross-contaminated.
Boss identity is keyed on biome strings inside BossSystem. if (g.Level.Biome == "hell") FistSlam(...) appears in Update and FirePattern. Adding a fifth biome boss means editing BossSystem in four places. The boss verb should be data on the boss block/level, not inferred from a biome string.
Trust boundary leak in /complete?treasureBonus=N. The client reports its own crystal bonus and the server adds it to the reward. The server already computes ItemTreasureBonus from the equipped items — the client value is pure cheat surface. More broadly: the sim knows when a level is won, but the meta trusts a bare client POST; nothing ties /complete to an actual completed session.
ProfileNs: any caller picks any profile via header/query with zero auth — fine for local dev, but it's an architectural decision that everything above it (stores, endpoints) now assumes.
Dual config-loading paths: Program.cs loads four catalogs once at startup, but GameSession.LoadLevel re-reads blocks.json/relics.json/bonuses.json from disk per websocket session. Two sources of truth for the same data, and the editor endpoints write into that same configRoot while the server runs.
Test scaffolding compiled into the production path: CheatHandler (with winNow, setLives, freezeMana) is reachable from any websocket client with no environment gate.

4. Naming inconsistencies

"Shield" means four unrelated things: the Paladin barrier spell (a Barrier), the ShieldStatue enemy behavior, the per-block ShieldTimer immunity, and powerup_shield (a ball auto-save). Same with "drain": the Necromancer spell, the drain line, and lava HP drain.
Hazard vs Projectile: two lists, two counters, one class, and one system that crosses them.
Necromant (block behavior) vs Necromancer (player character) — one letter apart, both live in damage-resolution code.
Economy currencies: Crystals, coins bonus, treasure effect, Points — three names map to one currency and one to another; ItemTreasureBonus grants crystals.
Lives doubles as HP (comment: "HP, enemy damage (M3+)") while SpareBalls/StartBalls is the actual classic-lives concept. Reader has to learn this from a comment in SimConfig.
Bonus vs Power-up: BonusSystem handles both with parallel configs (BonusDropChance/PowerUpDropChance, BonusEffectDuration/PowerUpWideDuration) and parallel effect branches — they're one mechanic with two vocabularies (visible "task 1.2" accretion).
Minor but telling: DungeonService.GenerateChoices rolls indices with (int)(rng.NextDouble()*count) + clamp while Rng.Range(int) exists and is used ten lines later in RiftService.

5. Missing separations of concern

Sim core knows the wire format. GameInstance stores List<EventDto> (a Net type) and Snapshot lives beside it. Events should be a domain type; mapping to DTOs belongs at the session boundary.
GameSession does five jobs: transport (websocket framing), content loading (re-reading catalogs), profile/dungeon buff application, the fixed-tick game loop, and serialization. Note the loop itself is wrong as a "fixed" timestep: await Send(...); await Task.Delay(dtMs) means tick rate = dt + send latency + scheduler jitter, so game time drifts from wall time and varies per client connection quality. A proper accumulator (Stopwatch-driven, tick N times per wake) is the standard fix.
No reward path from sim to meta. The server simulates the win (GamePhase.Won) but the session never grants rewards; the client must call /complete. The two halves of the server don't talk — that's both an integrity hole and a sign the session and the profile services need a shared application layer.
No DI / composition root discipline. Manual singletons passed into static Map(...) methods; stores hard-code AppContext.BaseDirectory/../../../saves, which breaks under dotnet publish layouts and is untestable without filesystem games. The handlers also each repeat load-mutate-save with no unit-of-work, so two concurrent endpoint calls on one profile do read-modify-write lost updates (the store lock protects file IO, not the transaction).
Logging policy in the wrong place: FileSimLog(verbose: true) per session unconditionally; verbosity is a deployment concern, currently a constructor literal.

6. Patterns that won't scale

O(n) entity scans everywhere, every tick. Grid exists but blocks are a flat List<Block> looked up by linear scan: ResolveBlocks iterates all blocks per ball per tick; SpreadFire does FirstOrDefault over all blocks per neighbor; Phoenix, zones, firewalls, shield statues, lava creep, and SeraphSummon's free-cell search each do full scans. A Dictionary<(col,row), Block> or 2D array maintained alongside the list turns nearly all of these into O(1). Related: dead blocks are never removed, so the list grows monotonically (descend, lava creep, overload bombs, seraph adds all append), and every scan pays for the corpses forever.
Full-state JSON snapshot at 60 Hz per session. Every tick allocates a complete Snapshot object graph, serializes all blocks/balls/hazards, and pushes the full string. With a 16×24 grid that's already hundreds of objects per frame per client; no delta encoding, no binary framing, heavy GC pressure. This is the first thing that falls over with concurrent sessions.
The legacy id-offset hack — projectiles appended into the Balls snapshot list with Id = 10000 + pr.Id — is a time bomb: _nextBallId is unbounded, and long sessions (or duplicate/split-heavy builds) will collide ball ids with projectile ids on the client.
File-per-profile JSON with an in-process lock as the persistence layer means single-node only, no concurrent writers, no migration story. Acceptable for a prototype; the issue is that the endpoint layer is written directly against these stores, so swapping in real persistence later means touching every handler. An IProfileRepository seam now is cheap.
One-block-per-tick collision with no substep: at BallSpeed × 1.4 escalation × WitchThrowSpeedMult, a ball moves ~8.4 units/tick — fine against 32-unit cells today, but every additive speed mechanic narrows the margin to tunneling, and there's no continuous-collision fallback.
Feature growth is additive in exactly the wrong places: each new mechanic = new fields on GameInstance + new branch in DamageBlock + new knobs in SimConfig + new fields in Snapshot + new string Kind. The marginal cost of feature N is proportional to N. The two structural moves that flatten that curve: (1) a block-death event pipeline replacing the DamageBlock switch-yard, and (2) per-feature state objects (components) on GameInstance instead of loose fields, with systems owning their own state.
